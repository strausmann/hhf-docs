Git Submodules vs. Forks\n\n**Empfehlung: Forks als Git Submodules**\n\nWorkflow:\n1. Fork `hhftechnology/crowdsec_manager` -> `strausmann/crowdsec_manager`\n2. Fork `hhftechnology/middleware-manager` -> `strausmann/middleware-manager`\n3. Beide Forks als Submodules in `projects/` einbinden:\n   ```\n   git submodule add git@github.com:strausmann/crowdsec_manager.git projects/crowdsec-manager\n   git submodule add git@github.com:strausmann/middleware-manager.git projects/middleware-manager\n   ```\n4. Arbeiten erfolgt in Feature-Branches innerhalb der Submodules\n5. PRs gehen von `strausmann/crowdsec_manager:feature-branch` -> `hhftechnology/crowdsec_manager:main`\n\n**Vorteile:**\n- Saubere Trennung: Eigenes Repo hat die Planung/Analyse/Staging, Submodules haben den Code\n- Submodule-Pointer trackt exakt welcher Stand gebaut wird\n- `git submodule update --remote` holt upstream-Aenderungen\n- Lokales Arbeiten in den Submodules fuehlt sich an wie ein normales Repo\n\n**Nachteile (akzeptabel):**\n- Submodule-Handling ist etwas umstaendlicher (Makefile-Targets helfen)\n- Clone braucht `--recurse-submodules`\n\n---\n\n### 7. CI/CD mit GitHub Actions\n\nZwei Workflows, analog zum bestehenden `docs-container-build.yml`:\n\n**build-crowdsec-docs.yml:**\n```yaml\non:\n  push:\n    branches: [main]\n    paths: ['projects/crowdsec-manager/docs/**']\n  workflow_dispatch:\n\n# Build mit Dockerfile.fumadocs\n# Push zu ghcr.io/strausmann/docs-crowdsec-manager:latest\n# Optional: Dockhand Hawser Webhook\n```\n\n**build-middleware-docs.yml:**\n```yaml\non:\n  push:\n    branches: [main]\n    paths: ['projects/middleware-manager/docs/**']\n  workflow_dispatch:\n\n# Build mit Dockerfile.fumadocs\n# Push zu ghcr.io/strausmann/docs-middleware-manager:latest\n# Optional: Dockhand Hawser Webhook\n```\n\n**pr-docs-check.yml:**\n```yaml\non:\n  pull_request:\n    paths: ['projects/*/docs/**']\n\n# npm ci + npm run build in Fumadocs-Verzeichnis\n# Validiert dass der Build nicht bricht\n```\n\n**Wichtig:** Submodules muessen in den Actions mit `submodules: recursive` ausgecheckt werden:\n```yaml\n- uses: actions/checkout@v4\n  with:\n    submodules: recursive\n```\n\n---\n\n### 10.
